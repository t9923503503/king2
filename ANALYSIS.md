# Анализ проекта: King of the Court (Лютые Пляжники !!)

## 1. Общая архитектура

Проект — **одностраничное PWA-приложение** для проведения пляжного волейбольного турнира в формате «Король площадки». Весь код — один файл `index.html` (~3 435 строк), включающий HTML-разметку, встроенный CSS и JavaScript без внешних фреймворков.

### Структура файла

| Блок | Строки | Назначение |
|------|--------|------------|
| CSS  | 1–1030 | Стили (переменные, навигация, корты, дивизионы, таймер, статистика) |
| HTML | 1031–1060 | Разметка (nav-заглушка, `#screens`, модалки, toast, кнопка scroll-top) |
| JS   | 1061–3435 | Вся бизнес-логика |

## 3. Бизнес-логика турнира

### Этап 1 — Корты
- До 4 кортов одновременно
- На каждом корте: **N игроков-мужчин × N раундов**, каждый раунд мужчина играет с конкретной женщиной по формуле `partnerW(mi, ri) = (mi + ri) % N`
- Очки за раунд: 0–15, хранятся в 3D-массиве `scores[court][man][round]`
- Очки женщин выводятся автоматически из очков партнёра-мужчины

### Этап 2 — Финалы (дивизионы)
- После Этапа 1 игроки ранжируются и распределяются по дивизионам: HARD (топ), ADVANCE, MEDIUM, LITE
- Та же схема раундов, отдельные массивы `divScores` / `divRoster`
- Комбинированный рейтинг = сумма очков Этап 1 + Финалы

## 4. Синхронизация (Supabase)

- Клиент: `@supabase/supabase-js` (CDN)
- Таблица: `kotc_sessions` с полями `room_code`, `state` (JSONB), `updated_at`
- **Подключение** — `sbConnect()`: pull текущего state, если сервер пуст — push локального
- **Push** — `sbPush()`: debounce 400 мс после каждого `saveState()`
- **Receive** — Postgres Realtime `UPDATE` событие → `sbApplyRemoteState()`
- **Smart merge** по timestamp-полям `scoreTs` и `timerTs` — выигрывает более новое состояние

## 7. Сильные стороны

1. **Нулевые зависимости** — работает полностью offline как PWA, нет сборщика
2. **XSS-защита** — функция `esc()` экранирует весь пользовательский ввод перед вставкой в innerHTML
3. **Smart merge при синхронизации** — конфликты данных между устройствами решаются по timestamp, не «кто последний»
4. **safeRender()** — при ре-рендере сохраняется позиция скролла и фокус активного поля ввода
5. **Рендер по требованию** — экраны рендерятся только при переключении вкладки, не держат устаревший DOM

## 8. Потенциальные проблемы и риски

### 8.1 Безопасность
- **Захардкоженный Supabase anon-key** (строка 2213): ключ вшит в исходный код, виден любому. Для `anon`-роли это допустимо, но предполагает правильно настроенный Row Level Security на стороне Supabase
- **Нет валидации room_code** на стороне клиента — пользователь может ввести произвольную строку; следует ограничить символы/длину

### 8.2 Архитектура
- **Монолит ~3 400 строк** в одном файле затрудняет поддержку; нет модульности
- **innerHTML везде** (~22 вхождения) — каждый ре-рендер пересоздаёт DOM-поддерево; при частых изменениях очков это неоптимально
- **Глобальные переменные** для всего состояния (scores, divScores, ppc, nc и т.д.) — отсутствует инкапсуляция

### 8.3 Логика
- `autoDistribute()` / `resetRosterNames()` жёстко ограничены 4 кортами (`ci < 4`), хотя интерфейс позволяет задать до 4 — ОК, но граничный случай не документирован
- `loadState()` использует строковое сравнение `ver < '1.1'` вместо семантического — при версии `'1.10'` сравнение сломается
- Свайп между кортами **отключён** (`attachSwipe()` пустой) — функциональность не реализована

### 8.4 UX
- При потере интернет-соединения Supabase-ошибки перехватываются тихо (`catch(e){}`), пользователь не всегда получает обратную связь
- Google Sheets OAuth flow требует popup — может быть заблокирован браузером

---

## 9. Итог

Проект — функциональное, хорошо структурированное приложение для конкретной задачи. Код читаем, содержит подробные комментарии на русском, логические секции разделены визуально. Основные риски — монолитная структура и захардкоженные credentials — типичны для hobby-проектов и не являются критическими при правильных настройках RLS. Приложение готово к использованию в текущем виде.
